# 音频测试报告全局解决方案

更新日期：2026-03-23 11:10:15

## 一、文档目的
这份文档的出发点不是先讨论技术实现，而是先明确：当前音频测试报告检查与 checklist 自动填写，究竟是在解决哪些业务问题。

因为这个项目本质上不是“做一个通用文档解析器”，而是围绕音频工程师的实际检查需求，逐步把人工检查流程产品化。所以文档结构也应按照这个顺序展开：

1. 先说明业务需求是什么。
2. 再说明基于这些需求，系统应采用什么总体思路。
3. 再说明当前项目已经做到哪里。
4. 最后说明后续最值得继续投入的方向。

## 二、业务需求

当前开发的直接出发点，是音频工程师对测试报告检查和 checklist 填写提出的一系列业务需求。这些需求可以先收敛为五组。

### 2.1 测试数据填写需求
1. 与 checklist 对照，没有多测。
2. 与 checklist 对照，没有漏测。
3. 与 checklist 对照，没有重测。
4. 测试数据应能正确填写到 checklist 对应位置。
5. 测试数据对应的判定结果也应正确。

### 2.2 测试内容合理性需求
1. 相同 codec 制式下，不同 network 的测试值应合理一致。
2. 相同 network 下，不同 codec 制式的测试值差异应合理。
3. 同一份报告中的 TX / RX 响度与频响趋势应基本一致。
4. 同一份报告中的相关曲线和数值之间应能互相印证，而不是彼此冲突。

### 2.3 测试时间检查需求
1. 相邻测试项的测试时间间隔应合理。
2. 同一份报告中不应出现异常长间隔的测试项。
3. 时延类测试时间应合理。
4. sidetone delay 测试时间应合理。
5. BGN connection 测试时间应合理。

### 2.4 报告内容与命名检查需求
1. 目录页数与报告总页数应能对应。
2. Word 报告中的 table / 章节应与目录对应。
3. 报告名称、页眉、Measurement Object 等关键信息应一致。
4. 测试报告名称应与实际测试项一致，包括 codec、bandwidth、terminal mode。
5. 报告名称中的 object 不应混入日期、测试版本等污染信息。
6. 报告中应能找到测试人员名字。
7. POLQA 相关算法库版本和参考音源设置应正确。

### 2.5 发布与更新体验需求
1. 桌面工具在安装后，应具备感知新版本的能力。
2. 当后续版本发布时，用户应能够在应用内检查更新，并支持在线升级安装版。
3. 更新能力应尽量不干扰核心检查、提取、回填主流程，而应作为交付与体验层能力独立建设。
4. 这项需求当前优先级靠后，应放在主功能稳定、核心流程跑顺之后，再进入持续优化阶段。

这五类需求里，当前项目已经覆盖得比较多的是“测试数据填写需求”；覆盖得较少的是“时间检查”和“Word 文档审查”。“发布与更新体验需求”已纳入整体规划，但当前不作为主线优先项。

## 三、基于需求的总体思路

如果从业务需求反推，当前系统不应该继续按“每来一种报告就加一段特殊规则”的方式扩张，而应该保持一个更稳定的总体方向。

### 3.1 保持 xlsx 作为主数值来源
1. 当前最稳定、最可验证的数值来源仍然是 xlsx。
2. checklist 自动填值主链应继续围绕 xlsx 提取来建设。
3. summary 和 detail 都要用，但应由同一条提取链统一协调，而不是拆成互相竞争的来源。

### 3.2 Word 更适合作为审查与证据来源
1. Word 当前更适合承担目录、命名、页眉页脚、人员信息、POLQA 配置、章节证据等检查任务。
2. 除非后续拿到更稳定的 Word 数值结构，否则不建议再让 Word 与 xlsx 争夺主数值来源。
3. 这样更符合当前项目已验证的样本质量差异，也能减少双源冲突。

### 3.3 提取规则和适用性规则应逐步拆开
1. 提取规则回答“报告里提到了什么、值是多少”。
2. 适用性规则回答“在当前 network、codec、bandwidth、terminal mode 下，本来应不应该测”。
3. 这意味着后续要逐步从 item 级规则里拆出独立的测试能力矩阵，而不是把所有语义继续塞在单条规则里。

### 3.4 系统最终不应只输出 checklist
1. checklist 自动填值只是第一层结果。
2. 业务上真正需要的，是一份更完整的结论：
   - 本次使用了哪些输入源。
   - 识别出的 codec、network、bandwidth、terminal mode、Measurement Object。
   - 哪些项命中、哪些项未命中、哪些项跳过。
   - 哪些项需要人工复核。
3. 所以后续应增加统一的结果输出层，而不是只停留在填表成功与否。

## 四、当前项目进展

这一部分建议按时间分段维护。后续如果有新的阶段性进展，直接在后面继续追加新的日期小节即可，这样比不断改写同一段正文更容易追踪项目演进。

### 4.1 2026-03-20：已落地且已验证的部分
1. 当前项目已经是可运行、可打包、可输出结果的 Electron + React 桌面工具，现有版本为 TN Audio Toolkit 1.0.0。
2. checklist 主链已经稳定在“xlsx 提取 -> checklist 回填 -> 模板样式尽量保留”的路径上。
3. checklist 写出主链已经切换为纯库静默执行，不再以 Office COM 作为主路径。
4. HA / HE / HH 已经不再适合共用一套强耦合模板假设，当前代码已开始按 mode-specific profile 收敛。
5. Report 页关键上下文字段已经有基础识别和写回能力，为后续参数确认与自动切 profile 提供了基础。
6. 本地 npm run build 已能成功产出 Windows 安装包与便携版，说明桌面工具交付链路已经可用。

### 4.2 2026-03-20：专项验证结果
1. HH / Handsfree 的 ambient 场景值已经能稳定写入正确目标单元格。
2. Handsfree 写回层旧的坐标二次改写已经移除，不再出现“提取成功但写错区域”的问题。
3. 当前 Handsfree 专项回归结果为：
   - HH-NB：50 / 68 命中，18 项 skip，0 项未命中。
   - HH-WB：51 / 68 命中，17 项 skip，0 项未命中。
   - HH-SWB：48 / 68 命中，17 项 skip，剩余 3 项未命中。
4. HH-SWB 剩余 3 项已确认是源报告缺值，不是规则未命中。
5. HE / Headset 近期已修正 outputCell 错位、sidetone 误判、delay 误吸附等问题，当前已回到纯 xlsx 提取主链。

### 4.3 2026-03-20：当前仍未完成的部分
1. 还没有正式的联合任务模型来统一 Excel、Word、checklist 三类输入。
2. 还没有真正的结论输出窗口，当前仍以 checklist 结果和命中明细为主。
3. 多测 / 漏测 / 重测 还没有形成独立、稳定的覆盖性审查引擎。
4. 跨报告一致性审查还没有产品化，仍偏脚本化和人工分析。
5. 时间合理性审查还没有统一时间线模型。
6. Word 侧目录、页码、页眉页脚、人员信息、POLQA 文档配置等审查能力仍未系统化落地。

### 4.4 2026-03-23：前端功能收敛与命名统一

## 五、后续建议

从业务价值和当前项目成熟度来看，接下来最值得做的，不是继续扩大规则数量，而是补齐几块真正缺失的结构能力。

### 5.1 第一优先级：统一任务模型
1. 前端一次任务应支持 Excel、Word、checklist 任意组合输入。
2. 后端应统一生成一份任务上下文，而不是让 Excel 和 Word 各跑各的。
3. 是否最终命名为 ReportBundle 可以后续再收口，但这层统一上下文应尽快正式化。

### 5.2 第二优先级：结论输出窗口
1. 第一版先不求大而全。
2. 先稳定输出以下信息：
   - 元数据识别结果。
   - 命中 / 未命中 / 跳过。
   - 是否缺少辅助输入源。
   - 建议人工复核项。
3. 这样前端才能真正从“自动填表工具”过渡到“审查辅助工具”。

### 5.3 第三优先级：独立适用性矩阵
1. 先把 bandwidth、vocoder、terminal mode 相关 skip 逻辑集中治理。
2. 再逐步扩到更多 network / codec 组合。
3. 目标是让“应测 / 不应测”的语义从 item 提取规则中独立出来。

### 5.4 当前不建议优先展开的方向
1. 为每种 codec / network / bandwidth 单独复制一套大规则。
2. 让 Word 与 xlsx 平级竞争主数值来源。
3. 在没有统一任务模型前，就大量铺开跨报告一致性分析。
4. 在没有结论输出层前，就堆很多零散审查点。
5. 一次性实现完整的长期理想架构。

### 5.5 后续体验优化方向：应用在线更新
1. 在线更新能力可以作为桌面工具交付体验的一部分纳入后续规划。
2. 该方向的目标不是改变当前报告检查主链，而是在版本迭代后降低用户手动下载安装新版本的成本。
3. 这项能力建议放在主功能、规则稳定性、结果输出和基础审查能力进一步成熟之后再落地。
4. 落地时优先支持 Windows 安装版的版本检测、在线下载和重启安装，不建议在当前阶段提前展开复杂的发布自动化体系。

## 七、结论

如果按当前项目状态和业务需求收敛成一句话，那就是：

这个项目应继续以音频工程师的业务检查需求为出发点，先稳住 xlsx 主提取链和 mode-specific profile，再逐步补齐统一任务模型、独立适用性矩阵和结论输出窗口，让系统从“能填 checklist”逐步走向“能输出基础审查结论”。
