# 音频测试报告全局解决方案

更新日期：2026-03-14

## 一、文档目的
本文档用于回答一个更高层的问题：

当测试报告同时覆盖不同 codec、不同 network、不同 bandwidth，并且不同场景下“该测哪些项、不测哪些项”并不一致时，系统应如何设计，才能比当前“直接基于规则从报告里提值”的方式更稳定、更可维护、更适合长期扩展。

本文档不再只讨论某一份报告怎么提取，而是从全局架构、需求拆解、模块边界、实施路径几个层面，给出一个更稳妥的方案。

## 二、先回答核心问题

### 1. 是否应该按每一种 codec / network / bandwidth 单独做一套规则？
不建议直接按“每一种情况一套规则文件”来做主方案。

这样做短期看起来直观，但长期会遇到几个明显问题：
1. 组合数会爆炸。
	一旦 codec、network、bandwidth、terminal mode、band、volume profile 叠加，规则数量会快速失控。
2. 相同逻辑会被重复实现。
	例如 loudness、sidetone、ambient、POLQA、formula 这些提取逻辑，本质并不会因为 codec 不同就完全重写。
3. 新报告一来，难以判断应该复用哪套规则。
4. 维护成本会逐步高于开发成本。

更好的做法不是“每种情况单独一套规则”，而是：

构建一个全局统一的解析框架，在框架内用“测试能力矩阵 + 提取计划 + 指标族提取器 + 少量家族化差异配置”来处理不同情况。

### 2. 是否应该完全忽略汇总表，只查测试详情？
也不建议。

正确做法不是“只用汇总表”或“只用详情页”，而是把两者都纳入统一中间层，再由系统按指标类型决定优先级。

更合理的分工是：
1. 汇总表负责：
	- 判断报告里测了哪些项；
	- 判断有没有多测、漏测、重测；
	- 提供测试项清单、状态和部分最终值；
	- 给详情页定位对应 section。
2. 详情页负责：
	- 提供更高精度原值；
	- 提供公式项的 margin、anchor、频点、场景值；
	- 提供汇总表里没有稳定保留的结构化结果。

所以更稳的方案一定是“汇总表 + 详情区协同”，而不是二选一。

## 三、推荐的总体思路

### 1. 全局目标
系统应该同时支持两类能力：
1. 数据提取能力。
	把报告里的测试结果稳定提取出来，并回填 checklist。
2. 报告审查能力。
	针对音频工程师提出的点检需求，自动发现“多测、漏测、重测、测试时间异常、命名不一致、目录异常、版本异常、交叉检查不合理”等问题。

当前系统更偏向第 1 类能力。后续全局方案应扩展为“提取 + 审查”双引擎。

### 2. 最优方案一句话概括
如果能拿到 ACQUA 原始结构化输出，就优先直接接原始数据；如果拿不到，就走“结构化文档解析 -> 统一中间层 -> 指标族提取 -> 审查规则引擎 -> checklist 回填 / 风险输出”的方案。

## 四、建议采用的系统框架

### 1. 第一层：报告接入层
负责把不同来源的报告转成统一输入。

支持来源可分为：
1. ACQUA 原始结构化结果文件，如 CSV、XML、数据库导出、中间结果包。
2. docx 报告。
3. legacy doc 报告。
4. checklist 模板。

这一层的职责不是提值，而是把“文件”转成“可被后续模块理解的数据源”。

### 2. 第二层：文档结构解析层
负责还原报告结构，而不是直接按文本搜关键词。

建议解析出这些结构：
1. report metadata
	例如 reportName、measurementObject、codec、network、bandwidth、terminal mode、测试日期、人员信息等。
2. summary sections
	汇总表中的结构化结果行。
3. detail sections
	测试详情中的 section 标题、段落、表格、锚点、状态。
4. section hierarchy
	例如 7.3.1a、8.3.1f 这样的测试编号与层级关系。
5. normalized numeric values
	对分裂小数、单位、状态值、band suffix、volume suffix 做标准化。

这一层产出的是“统一中间层”。

### 3. 第三层：统一中间层
这是整个方案最关键的一层。

中间层不是“只查详情”，也不是“直接等于汇总表”，而是把报告中的各类信息统一整理成程序内部稳定的数据模型。

建议至少包含以下对象：

1. ReportContext
	- reportId
	- reportName
	- codec
	- network
	- bandwidth
	- terminalMode
	- measurementObject
	- testDateRange

2. TestCaseSection
	- sectionId
	- sectionTitle
	- normalizedTitle
	- sourceRange
	- summaryRows
	- detailRows
	- tables
	- status

3. MeasurementEntry
	- metricType
	- rawValue
	- normalizedValue
	- unit
	- sourceKind
	- sourcePreview
	- confidence

4. ChecklistMappingContext
	- itemId
	- expectedMetric
	- preferredSource
	- formulaDependency
	- outputCell

5. AuditFinding
	- findingType
	- severity
	- message
	- relatedSection
	- evidence

后续所有提取、审查、回填逻辑都应只面对这套中间层，而不是直接面对原始 Word 文本。

### 4. 第四层：指标族提取引擎
不建议每个 checklist item 各写各的规则；更稳的做法是按指标族拆 extractor。

建议的提取器划分：
1. Loudness / Noise 提取器
2. Formula 提取器
	用于 idle channel noise、single frequency disturbances 等要基于标准值或基础项再计算的项目。
3. Ambient Noise 提取器
4. Sidetone 提取器
5. Sensitivity / Frequency Character 提取器
6. POLQA / P.863 提取器
7. Echo Control / DT-ST Class 提取器
8. Status / Pass-Fail 提取器

这样做的好处是：
1. 同类逻辑集中维护。
2. 不同 codec / network / bandwidth 之间主要复用同一类 extractor。
3. 真正差异只落在配置层，而不是每个 item 重写一遍。

### 5. 第五层：测试能力矩阵与提取计划层
这是最适合解决“不同场景下该测哪些项、不测哪些项”的地方。

建议单独维护一套“测试能力矩阵”或“提取计划配置”，而不是把这个问题塞进单条提取规则里。

这套矩阵回答的是：
1. 某种 network + codec + bandwidth + terminal mode 下，理论上应出现哪些测试项。
2. 哪些项是必须测。
3. 哪些项是可选项。
4. 哪些项在该场景下本来就不该测。
5. 某些项之间是否存在依赖关系。
6. 某些项跨 codec / network / bandwidth 时是否应该一致，还是应该呈现合理差异。

这层本质上就是“测试计划解释器”。

也就是说：
1. 提取引擎负责回答“报告里提到了什么、值是多少”。
2. 测试能力矩阵负责回答“在这种测试场景下，本来应该测什么、不应该测什么”。

这正好对应音频工程师图片里提出的很多需求。

### 6. 第六层：自动审查规则引擎
这一层不是为了提值，而是为了发现问题。

可分为四类审查：

1. 覆盖性审查
	对应工程师需求中的：
	- 没有多测；
	- 没有漏测；
	- 没有重测。

2. 一致性审查
	对应：
	- 相同 codec 不同 network 下测试值一致；
	- 相同 network 不同 codec 下测试值差值合理；
	- RX/TX 响度与频响趋势一致；
	- 同一份报告内部的相关曲线趋势一致。

3. 时间合理性审查
	对应：
	- 相邻测试项间隔时间合理；
	- 无异常长间隔；
	- 时延测试时间合理；
	- sidetone delay 测试时间合理；
	- BGN connection 测试时间合理。

4. 文档与模板完整性审查
	对应：
	- 目录页数与报告总页数对应；
	- word 报告 table 与目录对应；
	- 报告无异常回车、空白页；
	- 报告名称、页眉、measurement object 一致；
	- codec / 通话模式命名一致；
	- object 无日期和测试版本污染；
	- 报告有测试人员名字；
	- POLQA 算法版本和参考音源设置正确。

### 7. 第七层：checklist 回填与复核输出层
这一层不应只输出 Excel，还应输出审查结果。

建议输出至少两类结果：
1. checklist 填充值
2. 审查报告
	- 错误项
	- 警告项
	- 建议人工复核项
	- 每项的证据来源和 source preview

长期看，系统不应只是“自动填表工具”，更应成为“报告审查辅助工具”。

## 五、如何落地音频工程师提出的需求

### 1. 第一步：将测试数据填入汇总表
这类需求不只是提取值，更关键的是“覆盖性检查”。

建议做法：
1. 先用测试能力矩阵生成“本场景应测项清单”。
2. 再从报告 summary + detail 中识别“实测项清单”。
3. 做集合对比，得到：
	- 多测项
	- 漏测项
	- 重测项
4. 对相同 codec / network / bandwidth 的结果做分组比较，输出一致性检查结果。

### 2. 第二步：核对测试项时间
这类需求不能仅靠提值规则完成，必须在中间层保留测试时间元数据。

建议：
1. 解析每个 section 的测试开始/结束时间。
2. 构建测试时间线。
3. 为不同测试项配置合理的时间阈值和相邻间隔阈值。
4. 对特殊项单独建规则，例如 3quest、sidetone delay、BGN connection。

### 3. 第三步：核对 checklist 表
这类需求可以作为“提取结果与模板一致性校验”模块。

建议：
1. 对填充值做数值正确性检查。
2. 对判定值做状态正确性检查。
3. 对公式项保留公式来源、标准值、margin 值、计算结果，方便复核。

### 4. 第四步：其他审核
这部分应单独做“文档完整性与命名规范审查”模块，不应混在提值规则中。

建议拆成：
1. 目录与页数检查
2. section 与目录对应检查
3. 报告命名规范检查
4. object / measurement object 清洁度检查
5. 页眉页脚一致性检查
6. 人员信息存在性检查
7. 算法版本 / 参考音源配置检查

## 六、规则应如何重新分层

后续不建议继续只保留一份“大而平”的规则文件。建议把规则分成五层：

1. 全局归一化层
	处理 band、codec、suffix、数字、状态、单位的标准化。

2. 报告家族识别层
	判断当前报告属于 lamu 家族、3GPP Kansas 家族、Utah 家族，还是新的未知家族。

3. 指标族提取层
	按 ambient、POLQA、formula、sidetone、echo 等家族提取。

4. 测试能力矩阵层
	描述“在什么场景下应该测哪些项”。

5. 模板映射与回填层
	处理 checklist 的布局识别、outputCell 动态映射、格式设置、异常项说明。

## 七、为什么这个方案比“每种情况一套规则”更稳定
1. 差异被收敛到正确的层级。
	不是让每个 item 自己猜全文，而是把 codec / network / bandwidth 的差异留给测试能力矩阵和报告家族识别层处理。
2. 公共逻辑复用度更高。
	同一类项目的提取逻辑不会被重复拷贝 10 份。
3. 更容易扩样。
	新报告来时，优先判断是新增报告家族、还是新增测试矩阵、还是已有 extractor 的小变体，而不是立刻复制规则。
4. 更容易审计。
	每个提取结果、每个审查结论都可以追溯到来源 section 和证据。

## 八、实施建议

### 第一阶段：把当前系统升级成“中间层驱动”
目标：先解决当前直接面向文本的问题。

建议工作：
1. 继续增强 docx 结构化解析。
2. 在代码中显式定义统一中间层对象。
3. 让现有规则逐步迁移到中间层上运行。

### 第二阶段：建立测试能力矩阵
目标：让系统知道“什么场景该测什么”。

建议工作：
1. 先从已有样本反推矩阵。
2. 再让音频工程师确认矩阵是否完整。
3. 将矩阵配置化，而不是硬编码在规则里。

### 第三阶段：补齐自动审查引擎
目标：从“自动填表”扩展到“自动审查”。

建议工作：
1. 先做覆盖性审查。
2. 再做时间合理性审查。
3. 再做一致性与命名规范审查。

### 第四阶段：增加人工复核与异常闭环
目标：处理无法完全自动化的边界问题。

建议工作：
1. 输出低置信度项。
2. 输出标靶异常项。
3. 输出建议人工确认项。
4. 把人工确认结果沉淀为新的家族配置或矩阵规则。

## 九、最终建议
如果只给一个最稳妥的方向，建议采用下面这条路线：

不要为每一种 codec / network / bandwidth 单独复制一套规则；而是建立“结构化解析 + 统一中间层 + 指标族提取 + 测试能力矩阵 + 审查规则引擎 + checklist 回填”的整体框架。

如果能直接获取 ACQUA 原始结构化输出，就优先接原始数据；如果拿不到，再通过 docx 结构化解析构建中间层。这样才能既满足音频工程师的审查需求，也避免系统后续陷入规则爆炸和持续打补丁的状态。
