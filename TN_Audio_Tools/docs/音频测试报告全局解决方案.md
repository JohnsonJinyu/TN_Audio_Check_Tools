# 音频测试报告全局解决方案

更新日期：2026-03-20 15:18

## 一、文档目的
本文档用于回答一个更高层的问题：

当测试报告同时覆盖不同 codec、不同 network、不同 bandwidth，并且不同场景下“该测哪些项、不测哪些项”并不一致时，系统应如何设计，才能比当前“直接基于规则从报告里提值”的方式更稳定、更可维护、更适合长期扩展。

本文档不再只讨论某一份报告怎么提取，而是从全局架构、需求拆解、模块边界、实施路径几个层面，给出一个更稳妥、也更可执行的方案。

## 补充说明（2026-03-20）
本次更新不再把本文档只作为“理想架构草案”，而是明确补入截至当前代码基线的项目状态、已验证能力、未完成能力和近期代码提交记录。

换句话说，本文档现在同时回答两个问题：
1. 长期上，音频测试报告系统应如何设计。
2. 短中期上，当前项目已经做到哪里、下一步最值得做什么。

当前可明确确认的项目现状如下：
1. 当前产品已经是可运行的 Electron + React 桌面程序，包名版本为 `TN Audio Toolkit 1.0.0`，并已在本地通过 `npm run build` 成功产出 Windows 安装包与便携版。
2. checklist 填值主链路已经形成稳定的“结构化 Excel 提取 -> checklist 回填 -> 模板样式保留”的方案，且当前写出链路已切换为纯库静默执行，不再依赖 Office COM 作为主路径。
3. `HA / HE / HH` 三种 terminal mode 已经不再共用一套强耦合模板假设，而是逐步向 mode-specific profile 收敛；其中 `Handsfree` 与 `Headset` 最近一轮规则和坐标修正，已经验证可以显著减少误写与错位。
4. `xlsx` 目前是数值事实的主来源；`Word` 在现阶段仍更适合作为结构审查、辅助证据和兜底来源，而不是与 Excel 对等竞争数值主来源。
5. 当前系统已经具备一定的适用性判断能力，例如按 bandwidth、vocoder、terminal mode 跳过不测项；但“测试能力矩阵”还没有被彻底从规则层独立出来。
6. 当前还没有真正产品化的 `ReportBundle` 联合任务流，也没有完整的“结论输出窗口”；多测 / 漏测 / 重测、时间线审查、目录与命名审查仍处于方案明确但实现不完整的阶段。

## 二、先回答核心问题

### 1. 是否应该按每一种 codec / network / bandwidth 单独做一套规则？
不建议直接按“每一种情况一套规则文件”来做主方案。

这样做短期看起来直观，但长期会遇到几个明显问题：
1. 组合数会爆炸。
	一旦 codec、network、bandwidth、terminal mode、band、volume profile 叠加，规则数量会快速失控。
2. 相同逻辑会被重复实现。
	例如 loudness、sidetone、ambient、POLQA、formula 这些提取逻辑，本质并不会因为 codec 不同就完全重写。
3. 新报告一来，难以判断应该复用哪套规则。
4. 维护成本会逐步高于开发成本。

更好的做法不是“每种情况单独一套规则”，而是：

构建一个全局统一的解析框架，在框架内用“测试能力矩阵 + 提取计划 + 指标族提取器 + 少量家族化差异配置”来处理不同情况。

### 2. 是否应该完全忽略汇总表，只查测试详情？
也不建议。

正确做法不是“只用汇总表”或“只用详情页”，而是把两者都纳入统一中间层，再由系统按指标类型决定优先级。

更合理的分工是：
1. 汇总表负责：
	- 判断报告里测了哪些项；
	- 判断有没有多测、漏测、重测；
	- 提供测试项清单、状态和部分最终值；
	- 给详情页定位对应 section。
2. 详情页负责：
	- 提供更高精度原值；
	- 提供公式项的 margin、anchor、频点、场景值；
	- 提供汇总表里没有稳定保留的结构化结果。

所以更稳的方案一定是“汇总表 + 详情区协同”，而不是二选一。

### 3. 如果单独的 `.xlsx` 或 `.doc` 都无法覆盖全部信息，是否应该允许同时上传两份报告？
可以，而且这是当前阶段非常值得采用的折中方案。

更推荐的落地方向不是“二选一”，而是“联合输入、分工取长、统一输出结论”：
1. Excel 报告优先承担结构化数值提取。
	适合提取 checklist 填值、状态、bandwidth、部分时间字段、Measurement Object 等稳定结构化信息。
2. Word 报告优先承担版式与内容审查。
	适合检查目录、章节层级、页眉页脚、报告名称一致性、曲线图与说明块、测试人员信息、算法版本与参考音源设置等展示层内容。
3. 两者不必强行互相替代，而应在统一中间层中合并成同一个“报告包”。
	也就是前端允许一次任务上传：
	- 一个 Excel 报告；
	- 一个 Word 报告；
	- 一个 checklist 模板；
	系统再将两份报告按同一测试对象进行聚合分析。

这比“只押宝 Excel”或“只押宝 Word”都更稳，因为两种来源擅长的事实类型本来就不同。

### 4. 不同 codec / network / bandwidth 是否可以从文件名或内部字段识别？
可以，而且当前就应该把“文件名 + 内部字段”作为双来源元数据识别链路。

建议优先级如下：
1. 先读结构化字段。
	例如 Excel `Detailed` / `Values` 中的 `Bandwidth`、`Measurement Object`、`UseCase` 等列。
2. 再读报告内部文本。
	例如 Word/Docx 正文中的 section 标题、Measurement Object、codec/network/band 相关关键词。
3. 最后用文件名补齐。
	例如 `LAMU26_DVT1_VOLTE_AMR_WB_HA_0928` 这类命名中，通常已经携带：
	- network：`VOLTE`
	- codec：`AMR`
	- bandwidth：`WB`
	- terminal mode：`HA`

因此，更合理的实现不是只依赖某一处，而是把三类信息合并后生成统一的 `ReportContext`。

## 三、推荐的总体思路

### 1. 全局目标
系统应该同时支持两类能力：
1. 数据提取能力。
	把报告里的测试结果稳定提取出来，并回填 checklist。
2. 报告审查能力。
	针对音频工程师提出的点检需求，自动发现“多测、漏测、重测、测试时间异常、命名不一致、目录异常、版本异常、交叉检查不合理”等问题。

当前系统更偏向第 1 类能力。后续全局方案应扩展为“提取 + 审查”双引擎。

### 2. 最优方案一句话概括
如果能拿到 ACQUA 原始结构化输出，就优先直接接原始数据；如果拿不到，就走“结构化文档解析 -> 统一中间层 -> 指标族提取 -> 审查规则引擎 -> checklist 回填 / 风险输出”的方案。

### 3. 业务需求清单化
结合音频同事给出的点检项，当前全局需求可先明确拆成四组，这一清单后续可直接作为审查引擎与结论输出窗口的需求来源。

#### 3.1 第一步：将测试数据填入汇总表
1. 与 checklist 对照，没有多测。
2. 与 checklist 对照，没有漏测。
3. 与 checklist 对照，没有重测。
4. 相同 codec 制式下，不同网络的测试值一致。
5. 相同网络下，不同 codec 制式的测试值差值合理。
6. 不同网络 / 相同 codec 制式下，RX/TX 频响曲线趋势一致。
7. 同一份报告里的 TX/RX 响度与频响曲线趋势一致。
8. 同一份报告里的 RX 频响曲线与响度曲线低频幅值一致。

#### 3.2 第二步：核对测试项时间
1. 相邻测试项的测试间隔时间合理，除 3quest 等特殊项外，一般应在 1 分钟以内。
2. 同一份报告里不应出现异常长间隔的测试项，例如间隔一天以上。
3. 时延测试时间合理。
4. sidetone delay 测试时间合理。
5. BGN connection 测试时间合理。

#### 3.3 第三步：核对 checklist 表
1. 测试数据填写正确。
2. 测试数据判定正确。

#### 3.4 第四步：其他审核
1. 目录页数与报告总页数能够对应。
2. Word 报告中的 table 表要能和目录一一对应。
3. 测试项数量与目录要求能对应上，且目录已做全域更新。
4. 报告无异常回车键和空白页。
5. 报告名称与页眉和 Measurement / table 表中的 object 一致。
6. 测试报告名称与实际测试项名称一致，包括 codec 制式（NB/WB）与通话模式（HA/HE/HH）。
7. 报告名称中的 object 不应带日期和测试版本污染。
8. 报告里应存在测试人员名字。
9. POLQA 测试项的算法库版本和参考音源设置正确。

## 四、建议采用的系统框架

### 1. 第一层：报告接入层
负责把不同来源的报告转成统一输入。

支持来源可分为：
1. ACQUA 原始结构化结果文件，如 CSV、XML、数据库导出、中间结果包。
2. ACQUA 结构化 Excel 报告。
3. docx 报告。
4. legacy doc 报告。
5. checklist 模板。

这一层的职责不是提值，而是把“文件”转成“可被后续模块理解的数据源”。

更推荐的前端交互不是“单报告单次上传”，而是支持一个任务上传一个“报告包”：
1. 一个 Excel 报告，可选但强烈推荐。
2. 一个 Word 报告，可选但强烈推荐。
3. 一个 checklist 模板。

如果用户只提供其中一种报告，也应允许单源运行；但当 Excel 和 Word 同时存在时，系统应进入“联合分析模式”。

### 2. 第二层：文档结构解析层
负责还原报告结构，而不是直接按文本搜关键词。

建议解析出这些结构：
1. report metadata
	例如 reportName、measurementObject、codec、network、bandwidth、terminal mode、测试日期、人员信息等。
2. summary sections
	汇总表中的结构化结果行。
3. detail sections
	测试详情中的 section 标题、段落、表格、锚点、状态。
4. section hierarchy
	例如 7.3.1a、8.3.1f 这样的测试编号与层级关系。
5. normalized numeric values
	对分裂小数、单位、状态值、band suffix、volume suffix 做标准化。

这一层产出的是“统一中间层”。

这里需要特别强调：`report metadata` 不能只从一个地方读，而应做三路归并：
1. 文件名识别。
2. 结构化列识别，例如 Excel 的 `Measurement Object`、`Bandwidth`、`UseCase`。
3. 正文文本识别，例如 Word 章节标题、页眉页脚、详情块描述。

### 3. 第三层：统一中间层
这是整个方案最关键的一层。

中间层不是“只查详情”，也不是“直接等于汇总表”，而是把报告中的各类信息统一整理成程序内部稳定的数据模型。

建议至少包含以下对象：

1. ReportContext
	- reportId
	- reportName
	- codec
	- network
	- bandwidth
	- terminalMode
	- measurementObject
	- testDateRange

2. TestCaseSection
	- sectionId
	- sectionTitle
	- normalizedTitle
	- sourceRange
	- summaryRows
	- detailRows
	- tables
	- status

3. MeasurementEntry
	- metricType
	- rawValue
	- normalizedValue
	- unit
	- sourceKind
	- sourcePreview
	- confidence

4. ChecklistMappingContext
	- itemId
	- expectedMetric
	- preferredSource
	- formulaDependency
	- outputCell

5. AuditFinding
	- findingType
	- severity
	- message
	- relatedSection
	- evidence

6. ReportBundle
	- excelReport
	- wordReport
	- mergedContext
	- extractionResults
	- auditFindings
	- finalConclusion

后续所有提取、审查、回填逻辑都应只面对这套中间层，而不是直接面对原始 Word 文本。

### 4. 第四层：指标族提取引擎
不建议每个 checklist item 各写各的规则；更稳的做法是按指标族拆 extractor。

建议的提取器划分：
1. Loudness / Noise 提取器
2. Formula 提取器
	用于 idle channel noise、single frequency disturbances 等要基于标准值或基础项再计算的项目。
3. Ambient Noise 提取器
4. Sidetone 提取器
5. Sensitivity / Frequency Character 提取器
6. POLQA / P.863 提取器
7. Echo Control / DT-ST Class 提取器
8. Status / Pass-Fail 提取器

这样做的好处是：
1. 同类逻辑集中维护。
2. 不同 codec / network / bandwidth 之间主要复用同一类 extractor。
3. 真正差异只落在配置层，而不是每个 item 重写一遍。

### 5. 第五层：测试能力矩阵与提取计划层
这是最适合解决“不同场景下该测哪些项、不测哪些项”的地方。

建议单独维护一套“测试能力矩阵”或“提取计划配置”，而不是把这个问题塞进单条提取规则里。

这套矩阵回答的是：
1. 某种 network + codec + bandwidth + terminal mode 下，理论上应出现哪些测试项。
2. 哪些项是必须测。
3. 哪些项是可选项。
4. 哪些项在该场景下本来就不该测。
5. 某些项之间是否存在依赖关系。
6. 某些项跨 codec / network / bandwidth 时是否应该一致，还是应该呈现合理差异。

这层本质上就是“测试计划解释器”。

也就是说：
1. 提取引擎负责回答“报告里提到了什么、值是多少”。
2. 测试能力矩阵负责回答“在这种测试场景下，本来应该测什么、不应该测什么”。

这正好对应音频工程师图片里提出的很多需求。

### 6. 第六层：自动审查规则引擎
这一层不是为了提值，而是为了发现问题。

可分为四类审查：

1. 覆盖性审查
	对应工程师需求中的：
	- 没有多测；
	- 没有漏测；
	- 没有重测。

2. 一致性审查
	对应：
	- 相同 codec 不同 network 下测试值一致；
	- 相同 network 不同 codec 下测试值差值合理；
	- RX/TX 响度与频响趋势一致；
	- 同一份报告内部的相关曲线趋势一致。

3. 时间合理性审查
	对应：
	- 相邻测试项间隔时间合理；
	- 无异常长间隔；
	- 时延测试时间合理；
	- sidetone delay 测试时间合理；
	- BGN connection 测试时间合理。

4. 文档与模板完整性审查
	对应：
	- 目录页数与报告总页数对应；
	- word 报告 table 与目录对应；
	- 报告无异常回车、空白页；
	- 报告名称、页眉、measurement object 一致；
	- codec / 通话模式命名一致；
	- object 无日期和测试版本污染；
	- 报告有测试人员名字；
	- POLQA 算法版本和参考音源设置正确。

### 7. 第七层：checklist 回填与复核输出层
这一层不应只输出 Excel，还应输出审查结果。

建议输出至少两类结果：
1. checklist 填充值
2. 审查报告
	- 错误项
	- 警告项
	- 建议人工复核项
	- 每项的证据来源和 source preview

前端建议新增一个“结论输出窗口”或“结论面板”，不再只展示“命中多少项”，而是统一输出：
1. 本次使用的数据源。
	- 仅 Excel
	- 仅 Word
	- Excel + Word 联合分析
2. 元数据识别结果。
	- codec
	- network
	- bandwidth
	- terminal mode
	- Measurement Object
3. checklist 填写结果。
	- 命中项
	- 未命中项
	- 按场景跳过项
	- 低置信度项
4. 审查结论摘要。
	- 覆盖性问题
	- 时间异常
	- 文档命名或结构异常
	- 建议人工复核项
5. 证据链接。
	- 对应 section
	- source preview
	- 输出文件位置

长期看，系统不应只是“自动填表工具”，更应成为“报告审查辅助工具”。

## 五、如何落地音频工程师提出的需求

### 1. 第一步：将测试数据填入汇总表
这类需求不只是提取值，更关键的是“覆盖性检查”。

建议做法：
1. 先用测试能力矩阵生成“本场景应测项清单”。
2. 再从报告 summary + detail 中识别“实测项清单”。
3. 做集合对比，得到：
	- 多测项
	- 漏测项
	- 重测项
4. 对相同 codec / network / bandwidth 的结果做分组比较，输出一致性检查结果。

### 2. 第二步：核对测试项时间
这类需求不能仅靠提值规则完成，必须在中间层保留测试时间元数据。

建议：
1. 解析每个 section 的测试开始/结束时间。
2. 构建测试时间线。
3. 为不同测试项配置合理的时间阈值和相邻间隔阈值。
4. 对特殊项单独建规则，例如 3quest、sidetone delay、BGN connection。

### 3. 第三步：核对 checklist 表
这类需求可以作为“提取结果与模板一致性校验”模块。

建议：
1. 对填充值做数值正确性检查。
2. 对判定值做状态正确性检查。
3. 对公式项保留公式来源、标准值、margin 值、计算结果，方便复核。

### 4. 第四步：其他审核
这部分应单独做“文档完整性与命名规范审查”模块，不应混在提值规则中。

建议拆成：
1. 目录与页数检查
2. section 与目录对应检查
3. 报告命名规范检查
4. object / measurement object 清洁度检查
5. 页眉页脚一致性检查
6. 人员信息存在性检查
7. 算法版本 / 参考音源配置检查

## 六、规则应如何重新分层

后续不建议继续只保留一份“大而平”的规则文件。建议把规则分成五层：

1. 全局归一化层
	处理 band、codec、suffix、数字、状态、单位的标准化。

2. 报告家族识别层
	判断当前报告属于 lamu 家族、3GPP Kansas 家族、Utah 家族，还是新的未知家族。

3. 指标族提取层
	按 ambient、POLQA、formula、sidetone、echo 等家族提取。

4. 测试能力矩阵层
	描述“在什么场景下应该测哪些项”。

5. 模板映射与回填层
	处理 checklist 的布局识别、outputCell 动态映射、格式设置、异常项说明。

## 七、为什么这个方案比“每种情况一套规则”更稳定
1. 差异被收敛到正确的层级。
	不是让每个 item 自己猜全文，而是把 codec / network / bandwidth 的差异留给测试能力矩阵和报告家族识别层处理。
2. 公共逻辑复用度更高。
	同一类项目的提取逻辑不会被重复拷贝 10 份。
3. 更容易扩样。
	新报告来时，优先判断是新增报告家族、还是新增测试矩阵、还是已有 extractor 的小变体，而不是立刻复制规则。
4. 更容易审计。
	每个提取结果、每个审查结论都可以追溯到来源 section 和证据。

## 八、实施建议

### 0. 先给当前项目一个状态判断
为了避免后续执行时目标发散，当前状态建议明确为：

#### 已相对完整的部分
1. 单份报告到 checklist 的自动填值主链路已经基本可用。
	- Excel 报告已接入主方案；
	- Word 报告仍可兼容使用；
	- 公式项、ambient、echo、POLQA、layout 映射等已有较多落地基础。
2. 针对不同 bandwidth 的最小适用性规则已经开始落地。
	当前已支持部分 item 的 `applicableBandwidths`，并能在前端展示 `skippedItems`。
3. 部分报告上下文识别已经落地。
	当前已能从文件名、文本、Excel 结构列中识别 bandwidth，后续可继续扩展到 codec、network、terminal mode。
4. 输出侧基础能力较完整。
	包括 checklist 写入、百分比格式、边框、历史记录、输出文件管理等。

#### 当前仍明显不完整的部分
1. 多测 / 漏测 / 重测 的覆盖性审查还没有形成完整引擎。
2. 相同 codec / 不同 network 以及相同 network / 不同 codec 的横向对比还没有形成批量分析能力。
3. 时间合理性审查还没有形成统一时间线模型。
4. Word 侧的目录、页码、页眉页脚、table 与目录一致性等审查能力还没有正式实现。
5. 频响曲线趋势、响度与频响交叉关系，目前还没有进入结构化可审查状态。
6. 前端还没有真正的“结论输出窗口”，目前更多还是处理结果列表和明细查看。
7. Excel + Word 联合输入当前还没有产品化为一个正式任务流。

### 第一阶段：把当前系统升级成“中间层驱动”
目标：先解决当前直接面向文本的问题。

建议工作：
1. 继续增强 docx 结构化解析。
2. 在代码中显式定义统一中间层对象。
3. 让现有规则逐步迁移到中间层上运行。

### 第二阶段：建立测试能力矩阵
目标：让系统知道“什么场景该测什么”。

建议工作：
1. 先从已有样本反推矩阵。
2. 再让音频工程师确认矩阵是否完整。
3. 将矩阵配置化，而不是硬编码在规则里。

### 第三阶段：补齐自动审查引擎
目标：从“自动填表”扩展到“自动审查”。

建议工作：
1. 先做覆盖性审查。
2. 再做时间合理性审查。
3. 再做一致性与命名规范审查。

### 第四阶段：增加人工复核与异常闭环
目标：处理无法完全自动化的边界问题。

建议工作：
1. 输出低置信度项。
2. 输出标靶异常项。
3. 输出建议人工确认项。
4. 把人工确认结果沉淀为新的家族配置或矩阵规则。

### 第零优先级的下一步建议
如果按当前项目的实际收益排序，下一步最建议先做下面三件事，而不是立即把所有审查项一次性铺开：

1. 先定义“报告包”任务模型。
	让前端一次任务支持上传 Excel、Word、checklist 三类文件中的任意组合，并在后端生成统一的 `ReportBundle`。
2. 先实现“结论输出窗口”的最小版本。
	第一版不要求做全部点检，只先输出：
	- 识别出的 codec / network / bandwidth / terminal mode / Measurement Object；
	- checklist 命中 / 未命中 / 跳过；
	- 是否存在文件名与 Measurement Object 不一致；
	- 是否缺少 Word 或 Excel 辅助来源；
	- 建议人工复核的条目。
3. 再补第一批最容易落地的审查项。
	优先顺序建议为：
	- checklist 填写正确性与判定正确性；
	- 多测 / 漏测 / 重测；
	- 文件名、Measurement Object、bandwidth、codec、terminal mode 一致性；
	- 时间合理性中的基础间隔检查。

这条路线的好处是：
1. 不会卡在“频响曲线趋势”这类难度较高但短期难完全自动化的项目上；
2. 能先把 Excel 与 Word 的互补价值产品化；
3. 能更快让前端出现一个真正可交付的“结论反馈界面”。

## 九、最终建议
如果只给一个最稳妥的方向，建议采用下面这条路线：

不要为每一种 codec / network / bandwidth 单独复制一套规则；而是建立“结构化解析 + 统一中间层 + 指标族提取 + 测试能力矩阵 + 审查规则引擎 + checklist 回填”的整体框架。

在当前项目现实条件下，更推荐进一步收敛成这条可执行路线：
1. 能拿到 ACQUA 结构化 Excel 时，用 Excel 作为数值事实主来源。
2. 能同时拿到 Word 时，把 Word 作为文档结构、目录、曲线、命名和辅助证据来源。
3. 前端支持 Excel + Word + checklist 的“联合任务上传”。
4. 输出不再只有 checklist，还应新增“结论输出窗口”，把提取结果、适用性判断、审查摘要和人工复核建议统一反馈给用户。

这样既能保持当前已有能力继续复用，也更适合后续逐步把图中的点检项真正做成产品能力，而不是继续停留在单点提值工具阶段。


## 十、补充需求（重写版）

为确保本方案可直接进入开发执行，以下补充需求统一收敛为“任务流、交互、解析、输出、扩展”五个层面，并与前文的“报告包 + 中间层 + 指标族提取 + 审查引擎”保持一致。

### 10.1 任务输入与运行模式
1. 用户一次任务可上传三类文件中的任意组合：
	- ACQUA 导出的结构化 Excel 测试数据报告（`.xlsx`）。
	- ACQUA 生成的 Word 测试报告（`.doc` 或 `.docx`）。
	- checklist 模板（例如 Handset / Headset / Handsfree 版本）。
2. 运行模式按输入自动判定：
	- 仅 Excel：执行“数据提取 + checklist 回填”主链。
	- 仅 Word：执行“文档与审查”链路，不生成 checklist 填值结果。
	- Excel + Word：执行双链路联合分析，统一输出 checklist 结果与审查结论。
3. checklist 参考模板包括但不限于：
	- `Voice_Tuning_Checklist_v5.0.2_Handset.xlsx`
	- `Voice_Tuning_Checklist_v5.0.2_Headset.xlsx`
	- `Voice_Tuning_Checklist_v5.0.2_Handsfree.xlsx`

### 10.2 Report 页动态选择交互
1. checklist 的首个工作表为 `Report`，系统需读取并在前端展示以下可选项，供用户确认：
	- `B13`：Headset Interface
	- `B15`：Network
	- `C15`：Vocoder
	- `D15`：Bitrate
2. 前端交互要求：
	- 系统先解析模板中的可选值并渲染为下拉选择。
	- 用户确认后，系统按所选组合自动切换到对应测试工作表与规则 profile。
3. 业务约束：
	- 不同 `Vocoder` 对应的“应测/不测”项不同，适用性必须来自配置化规则（详见《需要测以及不需要测的规则》），不能写死在界面逻辑中。

### 10.3 主程序触发与链路分工
1. 用户完成文件上传与 Report 参数选择后，点击“开始解析报告”触发任务。
2. 链路分工如下：
	- Excel 链路：按提取规则和适用性规则提取值并写入 checklist。
	- Word 链路：执行目录、命名、页眉页脚、人员信息、POLQA 配置、章节证据等审查项。
	- 联合链路：同一 `ReportBundle` 中汇总两侧结果，输出统一结论。

### 10.4 静默执行与写出要求
1. 技术实现优先级：
	- 优先采用纯库方案静默执行（OOXML / xlsx 解析与定点写值等）。
	- 尽量避免依赖 Office COM 交互式通道，降低本地窗口弹出和误操作风险。
2. 输出要求：
	- 结果文件必须“另存为新文档”，不覆盖原模板。
	- 输出文档在结构、样式和版式上尽量保持与原 checklist 模板一致。
	- 跳过项、未命中项、低置信度项应在结果与结论层可区分。

### 10.5 多客户入口与规则治理
1. 在“开始解析”按钮附近新增“客户”下拉框，首批预设：
	- `SAMSUNG`
	- `MOTOROLA`（默认）
	- `T-Mobile`
	- `ATT`
2. 客户选择将驱动后端加载对应配置：
	- checklist 模板 profile
	- 规则 profile（提取规则 + 适用性规则）
	- 审查规则 profile
3. 治理原则：
	- 客户差异进入客户层配置。
	- HA / HE / HH 差异进入终端模式层配置。
	- 禁止继续通过 report-specific 临时代码分支维护长期规则差异。

1. 先把 `ReportBundle` 任务模型正式做出来。
	- 前端一次任务支持上传 `Excel / Word / checklist` 任意组合。

2. 再把“结论输出窗口”做成第一版。

	- 再展示命中 / 未命中 / 跳过 / 低置信度。
	- 最后补建议人工复核项和证据来源。
3. 把“适用性规则”继续从 item 提取规则中拆出去，形成独立的测试能力矩阵。
	- 当前项目已经有部分 skip 逻辑，但仍有不少适用性语义散落在规则文件里。
4. 把 Word 侧优先定位为“审查引擎输入”，而不是继续和 Excel 争夺主数值来源。
	- 这更符合当前项目已经验证过的实际数据质量差异。

## 十三、当前建议结论

结合当前项目状态，本文档的建议可以收敛为一句更务实的话：

当前系统不应继续沿“单条规则硬顶所有差异”的方向扩张，而应在保留现有 `xlsx` 主提取链可用性的基础上，逐步补齐四个关键层：
1. `ReportBundle` 联合任务模型；
2. 独立的测试能力矩阵；
3. 结论输出窗口；
4. Word 侧文档审查引擎。

这样既能保护当前已经稳定的 checklist 自动填值能力，也能让“报告审查辅助工具”这条路线真正开始落地，而不是继续把所有复杂度都压回到规则文件里。
