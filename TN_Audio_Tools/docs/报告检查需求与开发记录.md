# 报告检查需求、方案与开发记录

## 一、文档定位
本文档不再只是“功能需求清单”，而是同时承担以下三类信息沉淀：
1. 当前功能需求与验收口径；
2. 报告解析与 checklist 回填方案的设计演进；
3. 各阶段代码改动、验证结果与已确认结论。

## 二、功能背景
本功能用于从 ACQUA 音频测试报告中提取关键测试结果，并自动回填到 checklist 中，减少人工抄录工作量，提高测试报告检查效率。

## 三、基础功能需求

### 1. 输入与输出
1. 用户上传一个 ACQUA 导出的测试报告。
	当前优先支持结构化 Excel 报告，例如 `参考文件/UTAH/UTAH_HA_SWB.xlsx` 这类导出；同时继续兼容 Word 报告，例如 `docs` 目录下的 `Utah_Handset_mode_VOLTE_EVS_SWB_0521.doc`。
2. 用户上传一个 I 列数据区域为空的 Excel checklist，可参考 `docs` 目录下的 `moto_checlist_空.xlsx`。
3. 系统按照 `src/renderer/modules/reportChecker/config/moto_rules_for_analysis.json5` 中定义的数据提取规则，从报告中提取数据并写入 checklist。
4. 输出结果需尽量与标靶文件保持一致。当前参考标靶为 `docs` 目录下的 `Utah_Voice_Tuning_Checklist_v5.0.2_Handset_mode_VOLTE_EVS_SWB_0522_人工版本.xlsx`。

### 2. 规则来源说明
原始规则文件 `src/renderer/modules/reportChecker/config/moto_rules_for_analysis.json5` 最初是基于 `docs` 目录下的 `moto测试报告数据提取手写规则-0312更新.xlsx` 第 61 行及以上内容生成的。后续用户补充了 61 行以下的手写规则，因此需要继续补全剩余规则，并验证提取效果是否能做到与标靶文件一致。

## 四、注意事项
1. 某些 checklist 单元格的数据并不适合直接从汇总表取值，而需要从测试详情中提取。
2. 例如 `Handset` 页签中的 `I52`，目标值应来自测试详情标题 `Quality in the Presence of Ambient Noise AVG` 对应场景中的 `S-MOS` 值。从已有标靶来看，该值对应的是 `Crossroads` 场景表现。
3. 这类表格中的展示值可能经过四舍五入，而详情块中的值通常精度更高。当前实现优先允许使用详情中的原始值，必要时也可兼容汇总表值。

## 五、问题优化

### 1. 通用性问题描述
当前程序最初是以 `docs/Utah_Handset_mode_VOLTE_EVS_SWB_0521.doc` 为调试样本，并以 `docs/Utah_Voice_Tuning_Checklist_v5.0.2_Handset_mode_VOLTE_EVS_SWB_0521_人工版本.xlsx` 为标靶进行优化的。在该样本上，程序表现正常。

但在其他测试报告上验证时，发现程序存在明显的通用性问题，不能稳定提取不同报告中的对应数据，尤其是在 `参考文件/lamu_HA` 目录下的三份报告及其对应 checklist 标靶文件上，提取结果存在较多失败和错填情况。

因此，本轮优化目标是结合 `参考文件/lamu_HA` 下的三份测试报告与对应标靶文件，分析导致程序通用性不足的原因，并进行针对性修复。

## 六、解决方案记录

### 1. 问题定位结论
原有程序能够在 Utah 报告上工作，主要原因是当时的规则和解析逻辑对 Utah 报告的文本写法、表格结构和关键词组合适配较强。但在 lamu 系列报告上出现大量提取失败，说明问题并不是单个规则缺失，而是“规则 + 解析逻辑”整体对报告变体的适配能力不足。

经过比对，lamu 报告与 Utah 报告之间主要存在以下差异：
1. Band 标识不同，例如 `HANB`、`HAWB`、`HASB` 会混用。
2. 同一指标的标题写法不同，例如 `Sensitivity frequency charact.`、`Sensitivity, frequency character.`、`MOS-LQO P.863 informative` 等。
3. 同一条结果有的存在 HTML 表格结构，有的只存在于 rawText 单行文本中。
4. 某些指标在 Utah 报告中适合从汇总表提取，在 lamu 报告中则更适合从详情行或结构化结果行提取。
5. Ambient Noise 相关结果在详情区存在更高精度原值，而汇总表中的值可能已经做过四舍五入。
6. 回声控制、失真、单频干扰等项目在不同报告中的结果行与说明行拼接方式不同，旧逻辑容易发生串项。

### 2. 代码层面的主要调整

#### 2.1 文本归一化增强
修改文件：`src/main/services/reportCheckerService.js`

在 `normalizeText` 中补充了更多同义归一化规则，用于统一不同报告中的等价表达，主要包括：
1. 将 `HANB`、`HAWB`、`HASB` 等统一映射到同一类 Band 概念。
2. 将 `MXX-4` 这类异常写法归一为 `MAX-4`。
3. 将 `Receiving`、`Receiving Direction`、`receive` 等统一为 `RCV`。
4. 将 `Sending` 统一为 `SND`。
5. 将 `frequency`、`charact.`、`average`、`nominal`、`volume`、`mandatory` 等常见变体统一处理。

这样做的目的，是先把“同义但不同写法”的文本压平，再进行规则匹配，减少因措辞差异导致的漏提取。

#### 2.2 rawText 行级候选增强与收紧
修改文件：`src/main/services/reportCheckerService.js`

在 `createSearchData` 中新增了 raw line 候选行，用于覆盖那些没有稳定 HTML 表格结构、但在原始文本中能看到完整结果的场景。同时对 raw line 候选进行了收紧，避免把目录索引行、说明行、页面残片误当成可提取结果行。

当前筛选原则为：
1. 文本长度不能过短。
2. 必须带有显式状态信息，例如 `Ok`、`Not Ok`、`Done`，或者具备足够结构化的 cell 数量。

这项调整既支持了 legacy `.doc` 报告提取，也显著降低了错误命中概率。

#### 2.3 后缀判断逻辑优化
修改文件：`src/main/services/reportCheckerService.js`

在 `hasRequiredSuffixes` 中增强了 `MIN` 相关判断。旧逻辑中，像 `Min. dist. to tolerance scheme` 这样的文本，容易被误判成 `MIN` 音量档位，导致错误填值。新逻辑显式排除了 `Min. dist.`、`minimal` 等非音量档位语义，只保留真正代表 `MIN` 档位的匹配。

#### 2.4 单行文本描述段截取优化
修改文件：`src/main/services/reportCheckerService.js`

在 `getRowDescriptorText` 中增加了对 `Ok`、`Not Ok`、`Done` 之前文本的截断处理，保证规则匹配、后缀判断、黑白名单判断尽量只基于“项目描述部分”进行，而不是把结果值和状态字段一起算进去。这一点对 lamu 报告里大量“单行拼接结果”尤其重要。

#### 2.5 候选行评分机制优化
修改文件：`src/main/services/reportCheckerService.js`

新增统一的 `getCandidateScore` 评分机制，用于在多个候选结果中挑选更可信的一行。评分时综合考虑：
1. 是否具备结构化 cells。
2. 是否包含有效数字单元格。
3. 是否存在表头信息。
4. 是否包含明确状态值。
5. 文本长度是否更像完整结果行。

这样可以减少“说明行压过结果行”的情况，提高复杂报告下的稳定性。

#### 2.6 锚点提取逻辑优化
修改文件：`src/main/services/reportCheckerService.js`

对 `detail_anchor_extract` 类型增加了更稳妥的窗口选择和 fallback 逻辑。当锚点附近文本结构不稳定时，会回退到更适合的行匹配提取方式，而不是直接失败。同时对锚点候选窗口增加关键词得分排序，优先选择更接近目标条目的文本区块。

#### 2.7 regex 提取逻辑优化
修改文件：`src/main/services/reportCheckerService.js`

主要调整如下：
1. 优先从结构化结果行中直接做 regex 提取。
2. 如果结构化行无法提取，再退回到上下文窗口扫描。
3. 对多候选命中的情况不再直接取第一条，而是做评分后选择最合适的一条。
4. 对 requiredSuffix 的判断增加“同段、就近”约束，避免 short / long、1o2 / 2o2 互相串用。

这部分主要用于 Echo Control 的 DT/ST Class 相关项目。

#### 2.8 公式型提取逻辑优化
修改文件：`src/main/services/reportCheckerService.js`

对 `formula_calc` 类型增加了 `baseItemId` 支持。即某些公式项不再只依赖固定标准值与 margin，而是可以先引用前一个基础项的真实提取值，再进行计算。这主要用于 Single Frequency Disturbances 等项目，提升了不同报告结构下的公式兼容性。

#### 2.9 Ambient Noise 详情块提取优化
修改文件：`src/main/services/reportCheckerService.js`

将原先较依赖固定正则块的提取方式，改成按场景逐行跟踪解析，主要优化点包括：
1. 兼容 `TS 103 106` 与 `TS 103 281` 两类写法。
2. 优先从详情块读取更高精度的 `G-MOS`、`N-MOS`、`S-MOS` 原值。
3. 对 `N-MOS` 的提取显式排除了 `Corrected N-MOS`，避免误用校正值。
4. 对场景名统一做标准化，避免 `Road` 与 `Crossroad` 互相误命中。

### 3. 规则文件层面的主要调整

#### 3.1 规则调整总体思路
修改文件：`src/renderer/modules/reportChecker/config/moto_rules_for_analysis.json5`

规则文件的调整不是单纯放宽，而是“该放宽的放宽、该收紧的收紧”。总体策略如下：
1. 对不同报告里同义但不同写法的条目，增加容错能力。
2. 对容易串项的条目，增加精确限定。
3. 对需要公式联动的条目，补齐基础项依赖。

#### 3.2 提高部分规则的兼容性
去掉了部分只适合 Utah 报告的过强限定词，例如某些条目中对 `Speech`、`Volume`、`100 Hz-16000 Hz` 等词的依赖。同时为部分条目增加了 `fallbackKeywords`，使其在标准写法匹配不到时，能够接受等价写法。

这类条目主要包括：
1. Single Frequency Disturbances 相关项。
2. Sensitivity / frequency character 相关项。
3. Ambient Noise 的 G-MOS average 项。
4. P.863 / MOS-LQO informative 相关项。

这部分改动提高了规则对 lamu 报告的兼容性。

#### 3.3 对高风险串项条目增加精确约束
为部分条目增加了 `exactRowKeywords`，用于强制要求结果行必须包含某些关键语义词。典型例子如下：
1. `1/3oct` 项必须明确带 `1/3oct`。
2. 发送失真 `-4.7dBPa` 项必须明确带 `-4.7` 和 `dBPa`。
3. Echo Control 的 short 项必须明确带 `short`、`1o2`。
4. Echo Control 的 long 项必须明确带 `long`、`2o2`。

这部分调整的目的，是避免结果行被相邻项目串填。

#### 3.4 Echo Control 的 regex 规则增强
对 `DT Class A1`、`DT Class A1+A2`、`ST Class A1+A2` 等条目的正则表达式进行了增强，使其兼容空格、百分号、`[%]` 标记等不同排版形式。同时配合 `exactRowKeywords` 和 `requiredSuffix`，确保 short / long 两组结果不会混用。

#### 3.5 公式项规则定义增强
为 Single Frequency Disturbances 的发送端和接收端规则增加了 `baseItemId`，从而将当前条目的计算逻辑与前一个基础噪声项绑定，避免在不同报告结构下出现计算口径漂移。

#### 4.2 验证结果
1. lamu 系列样本在优化前，存在大量项目无法匹配或错误匹配，初始状态仅能正确命中少量条目。
2. 优化后，lamu 的 NB 基准比对结果已收敛到 0 差异，说明本轮“通用性问题”已被显著修复。
3. Utah 样本仍保留少量差异。经排查，这些差异主要不是由本轮通用性改动引起，而是原始人工标靶与报告原值之间本身存在口径差异，例如：
	- 某些 Utah 单元格标靶中保留空白，但报告中存在可提取结果。
	- 某些数值在人工版中采用了不同口径或抄写方式。

因此，本轮优化的核心目标，即“让程序能够更通用地处理不同测试报告”，已经达到预期。

### 5. 本轮改善思路总结
本轮优化不是通过简单扩大模糊匹配范围来解决问题，而是采用“解析能力增强 + 规则精准分层”的方式处理，整体思路如下：
1. 先增强底层文本归一化和候选行构造能力，让程序能够理解不同报告格式。
2. 再通过 `fallbackKeywords` 提升等价表达下的兼容性。
3. 最后通过 `exactRowKeywords`、suffix 判断、候选评分等机制，把容易串项的高风险条目重新收紧。

这种做法的收益是：
1. 提高了不同测试报告之间的通用性。
2. 避免因为一味放宽规则而造成大面积误填。
3. 便于后续继续补充新报告样本时做增量优化。

#### 8.7 `lamu_HA` 与 `3GPP Handset report` 差异总结
1. 两类语料的最大差异，不是单个关键词写法不同，而是“报告家族一致性”差别很大。
	`lamu_HA` 目录下的几份报告整体仍属于同一类报告家族，结构、标题组织方式、summary 与详情的对应关系比较接近；而 `3GPP Handset report` 目录下的 9 份报告同时覆盖了 `AMR_NB / AMR_WB / EVS_NB / EVS_WB / VoIP / VoNR / WCDMA` 等多种模式，报告之间的结构差异明显更大。
2. `lamu_HA` 暴露出来的主要还是规则兼容性问题，而 `3GPP Handset report` 暴露的是解析层、模板层和标靶层的系统性问题。
	在 `lamu_HA` 中，更多问题可以通过文本归一化、suffix 判断优化、候选行评分和规则收紧解决；而在 `3GPP Handset report` 中，除了规则问题，还同时存在：
	- 同一项在不同报告里最可信的数据来源不同；
	- 仅依赖 mammoth 的 html / rawText 无法稳定承载复杂结构；
	- Handset checklist 模板布局存在整体错位；
	- 部分标靶文件本身存在口径不一致问题。
3. `lamu_HA` 的问题更像“局部泛化”，`3GPP Handset report` 的问题更像“全局泛化”。
	`lamu_HA` 的目标是让同一家族的几份变体报告稳定提取；`3GPP Handset report` 则要求程序在跨 codec、跨 band、跨报告排版、跨 checklist 模板的情况下都尽量稳定工作，因此对程序泛化能力要求高得多。
4. 为什么程序处理 `lamu_HA` 没什么问题，但处理 `3GPP Handset report` 一开始问题很多，本质原因在于：
	- `lamu_HA` 优化后，大多数问题都能被归结为“规则不够兼容或不够收紧”；
	- `3GPP Handset report` 中大量问题不能只靠补关键词解决，而必须先升级底层解析架构，例如引入 docx OOXML 结构化解析、分离 `tableRows` / `fallbackRows`、修复值列表头误命中、处理分裂小数、增加 compact 3GPP checklist 布局识别。
5. 因此，`lamu_HA` 的成功更多说明“规则层已经足够适配这一类报告”，而 `3GPP Handset report` 的问题集中爆发，说明旧方案对更复杂语料的根本短板在于：
	- 结构化输入不够稳；
	- 候选行来源混杂；
	- 输出模板映射不够动态；
	- 标靶异常与解析异常没有被区分开。
6. 当前阶段的优化方向已经从“继续为某一份报告打补丁”，转为“提升对整类 3GPP 报告的解析能力”。这也是为什么在 `3GPP Handset report` 语料上，本轮工作重点不再只是改规则，而是同步改造 `reportSource`、`reportAnalysis`、`checklistLayout` 和回归分析脚本。

#### 8.8 面向多 codec / 多网络 / 多带宽报告的后续架构优化建议
1. 如果测试链路能够提供 ACQUA 的原始结构化结果，例如 CSV、XML、数据库导出或其它中间结果文件，应优先接入原始数据源，而不是继续把 Word 报告当作唯一事实来源。
	原因是 Word 报告本质上属于展示层，天然会引入表格变形、换行丢失、数值断裂、描述列和值列混淆等问题；如果能直接读取原始结果，稳定性、可解释性和字段映射精度都会显著高于基于报告文本的提取。
2. 在无法直接拿到原始数据源的前提下，最优方案不是继续堆叠关键词规则，而是采用“结构化文档解析 + 统一中间模型 + 指标族提取”的分层方案。
	建议将报告先解析为统一中间结构，例如：
	- section / test case 边界；
	- summary rows；
	- detail rows；
	- 结构化 tables；
	- band、codec、network、bandwidth 等上下文元数据；
	然后再由各类指标提取器在该中间结构上取值，而不是每个 item 直接面向原始文本全局搜关键词。
3. 相较于“每个 checklist item 独立配规则”，更推荐按指标族组织提取逻辑。
	例如可拆分为：
	- loudness / noise 类提取器；
	- sidetone 类提取器；
	- ambient noise 类提取器；
	- POLQA / P.863 类提取器；
	- formula 公式类提取器；
	- echo control / DT-ST class 类提取器。
	这样相同类型的字段判定、数值清洗、候选排序逻辑只维护一次，更适合面对多 codec、多网络、多带宽下的报告变体。
4. 可增加“报告家族识别”或“模板指纹识别”层，而不是假设所有报告共用一套同权规则。
	可利用标题样式、section 编号模式、Measurement Object 命名方式、典型表头组合、checklist Handset 页布局特征等信息，对报告先做家族分类，再选择对应的 extractor profile。这样既比“每来一份报告写特判”更可维护，也比“所有报告硬套同一组规则”更稳定。
5. 不建议直接把大模型或通用 AI 阅读整份报告后直接填 checklist，至少不应把它作为主提取链路。
	更合理的用法是：
	- 用于标题归一化；
	- 用于 section 分类；
	- 用于候选行排序或异常解释；
	- 用于辅助识别相似项名。
	主事实抽取链路仍应保持可复现、可回归、可审计的结构化解析与规则逻辑。
6. 后续回归体系建议显式区分三类问题，而不是全部混为“提取错误”：
	- extraction error：解析或提取逻辑错误；
	- template mapping error：checklist 模板布局或 output cell 映射错误；
	- target anomaly：标靶文件本身存在口径或数值异常。
	这样才能避免把标靶问题误当成规则问题，导致错误优化方向。
7. 综上，后续更优的实现路线应是：
	“原始结构化数据优先；拿不到原始数据时，走 docx 结构化解析优先；在统一中间结构上按指标族提取；用少量报告家族化规则做差异收口；模型仅作为辅助判别层。”
	这条路线比继续单纯扩大模糊规则范围，更适合长期支持多 codec、多网络、多带宽的测试报告。

##### 14.4.1 Excel 覆盖性口径
本轮将 Excel 审查聚焦为以下几类结论：
1. `matchedCount`：成功写入 checklist 的项目数；
2. `missingCount`：未命中的项目数，即当前重点关注的漏测项；

4. `duplicateCount`：同类结构化结果重复出现的重测候选；
5. `extraCandidateCount`：Excel 结构化结果里存在、但未进入 checklist 回填语料的多测候选。

其中“多测”当前仍定义为候选提示，而不是最终判定。原因是：
1. 仅靠 Excel 差集还不能完整表达测试计划口径；
2. 后续仍需与音频工程师确认哪些确实属于不该测却测了。

4. 报告名与页眉页脚一致性；
5. Measurement Object 清洁度检查，识别是否混入日期、版本、DVT 信息；
6. 测试人员 / 责任人字段识别；
7. POLQA / P.863 版本与参考音源配置识别；
8. 响度章节识别；
9. 频响章节识别。

其中响度与频响当前仍定位为“证据入口”而不是“自动判定正确性”，即：
1. 程序会指出 Word 中是否识别到相关章节；
2. 会把证据文本回传到结论窗口；
3. 最终正确性仍需音频工程师人工审核。

本轮新增一致性结论的触发逻辑：
1. 当存在“同 codec、不同 network”的多份 Excel 报告时，做跨 network 对比；

3. 对同一 checklist output cell 的实测值进行 spread 对比，差值偏大时标记为 `warning` 或 `review`。

这一层当前主要用于辅助发现：
1. 同类样本间数值偏差异常；
2. 文本值或状态值不一致；
3. 需要在结论窗口优先人工复核的差异项。

### 15. 2026-03-17 输出边框、I29 来源与 Utah_All 多模式专项记录

#### 15.1 本轮问题来源

#### 15.2 本轮核查结论

##### 15.2.1 边框异常的直接原因
当前运行时优先使用 `src/main/services/reportChecker/applyChecklistStyles.ps1` 通过 Excel/WPS COM 落样式。

此前 COM 脚本对整块 `A3:K75` 使用了：
1. 先对整块 `Range` 统一铺细线；
2. 再把四周外边框提成粗线。

这种做法在 checklist 模板存在大量 merge 的情况下并不稳定。经检查，当前空模板与 Utah_All 三个 checklist 模板在 `A3:K75` 内均存在大量 merge，例如：
1. `A3:D3`
2. `E3:H3`
3. `A5:K5`
4. `A6:A15`
5. `B7:B14`
6. `C7:C14`

因此，问题不在于“边框规格定义错了”，而在于“对整块 merge 模板直接设 Range 边框”会在部分内部区域产生异常粗线，用户看到的 `I/J/K` 列局部粗线，属于该实现方式的副作用。

##### 15.2.2 `I29` 当前提值来源核查结论
当前规则中，`I29` 对应 item 24：
1. `checklistDesc = Sidetone characteristics - Delay`
2. `outputCell = I29`
3. `extractType = summary_table_match`

对 Utah Handset Excel 样本实际回放后，当前程序抓取到的来源为 `Delay (Cross)` 行，且来源文本明确来自 sidetone delay 段，例如：
1. `7.5.4 Sidetone delay, NOM HANB | Delay (Cross) | ... | 2.24999687634408 | ms | Done`
2. `8.5.4 Sidetone delay, NOM HAWB | Delay (Cross) | ... | 2.24999687634408 | ms | Done`
3. `9.5.4 Sidetone delay, Volume NOM HASB | Delay (Cross) | ... | 2.24999687634408 | ms | Done`

因此：
1. 在 Handset 样本上，`I29` 并不是凭空生成的；
2. 它来自报告中的 `Delay (Cross)` 实测值；
3. 这个来源与 `Sidetone characteristics - Delay` 的测项语义是一致的。

但继续核查 `Utah_All/HE/UTAH_HE_SWB.xlsx` 后发现，Handsfree 模式下 item 24 曾被错误命中到：

这说明 item 24 在非 Handset 模式下存在误命中风险，之前并不能保证一定抓到真正的 delay 行。

1. Noise 11-15；
2. Sensitivity 16-20；
3. STMR 21-23；
4. Stability 25；
5. Ambient noise 部分项；

本轮确认的主要根因有两类：
1. 规则和文本归一化长期以 Handset 为主，许多 band 语义只兼容 `HANB / HAWB / HASB`，未覆盖 `HENB / HEWB / HESB` 与 `HHNB / HHWB / HHSB`；

### 16. 2026-03-17 模式规则文件拆分实施记录

#### 16.1 变更背景
在继续核查 `Utah_All` 目录下的 Handset / Handsfree / Headset 样本后，已确认三个模式不仅报告写法不同，连 checklist 的测项数量与 I 列落点也不完全一致。

在这种前提下，继续把三种模式硬塞进同一个规则文件，会导致：
1. `outputCell` 越来越依赖 mode-specific 特判；
2. 规则关键词既要照顾 HA，又要避免误伤 HE / HH；
3. 后续规则维护会越来越接近“在一个大文件里不断堆 if”。

因此，本轮正式将规则入口改为“按终端模式分 profile 加载”，而不是继续只维护一个 Handset 中心化的大规则文件。

#### 16.2 本轮实现方案

##### 16.2.1 默认规则入口改为 manifest
修改文件：`src/renderer/modules/reportChecker/config/moto_rules_for_analysis.json5`

当前该文件不再直接承载完整 `extractItemList`，而是改为 manifest 入口，只声明：
1. 默认 profile；
2. Handset / Handsfree / Headset 三个 profile 的规则文件路径；
3. 每个 profile 对应的 terminal mode。

当前 manifest 中的映射关系为：
1. `HA -> handset`
2. `HE / HF -> handsfree`
3. `HH / HS -> headset`

##### 16.2.2 新增三份独立规则文件
新增文件：
1. `src/renderer/modules/reportChecker/config/moto_rules_handset.json5`
2. `src/renderer/modules/reportChecker/config/moto_rules_handsfree.json5`
3. `src/renderer/modules/reportChecker/config/moto_rules_headset.json5`

本轮先以当前 Handset 规则为基线复制出三份 profile 文件，并在 `ruleBaseInfo` 中标记各自适用模式。这样做的目的不是宣称 HE / HH 已完全适配，而是先把“规则文件分治”的结构基础搭起来，后续 HE / HH 可以在自己的 profile 文件中独立演进，不再与 Handset 混在一起。

##### 16.2.3 规则加载链路支持 bundle 与单文件双兼容
修改文件：
1. `src/main/services/reportChecker/reportSource.js`
2. `src/main/services/reportChecker/reportExtractor.js`
3. `src/main/services/reportCheckerService.js`

本轮将规则加载能力升级为同时支持两种输入：
1. 旧式单文件规则：文件根节点直接包含 `extractItemList`；
2. 新式 bundle 规则：入口文件包含 `ruleProfiles`，由程序按 profile 自动选择具体规则。

选择逻辑当前基于 `reportContext.terminalMode`，并带 checklist 文件名兜底。这样可以保证：
1. 默认情况下，HA / HE / HH 报告自动进入各自 profile；

修改文件：`src/main/main.js`

此前“导出规则”逻辑是直接复制规则文件本体。如果默认规则文件变成 manifest，那么仅复制入口文件会丢失真实 profile 内容。

因此本轮同步修改为：
1. 若导出的是旧单文件规则，则仍按原文件内容导出；
2. 若导出的是新 manifest 规则入口，则在导出时自动展开并输出为一个完整的规则内容文件。

这样可以保证导出的规则文件仍可直接被查看和二次维护，而不是只拿到一个带相对路径引用的壳文件。

本轮已完成以下验证：

3. 直接指定单文件规则路径时，旧流程仍兼容可用；
4. 导出逻辑已能生成展开后的完整规则内容，避免 manifest 丢 profile。

当前实测结果示例：
1. `HA` 样本返回 `ruleProfileKey = handset`
2. `HE` 样本返回 `ruleProfileKey = handsfree`
3. `HH` 样本返回 `ruleProfileKey = headset`

### 17. 260318问题梳理

#### 17.1 当前确认的问题

##### 17.1.1 checklist 样式与版式仍需按模式拆分
当前已经确认 HA、HE、HH 三种模式的 checklist 版式并不一致，因此输出样式不能继续按同一块区域统一处理，而应按模板 profile 精确落样：
1. Handset：边框区域为 `A3:K75`，外围粗线，内部细线。
2. Handsfree：边框区域为 `A3:K57`，外围粗线，内部细线。
3. Headset：边框区域为 `A3:K50`，外围粗线，内部细线。

本问题的本质不是“边框粗细没设对”，而是“不同模式的有效内容区域不同，程序不应再对超出模板有效区的空白区域补多余框线”。

除边框外，模板中的结构行样式也需要按模式单独维护，不能继续沿用 Handset 的整页假设。当前已明确：
1. HE 模板只有第 `5`、`53` 行需要 `A:K` 合并、左对齐、深灰底色。
2. 后续 HA / HE / HH 的 merge、填充色、对齐方式都应由模板样式配置驱动，而不是在通用代码里写死。

##### 17.1.2 不测项需要有明确的“跳过 + 灰填充”语义
此前程序虽然已支持部分 `skip` 语义，但当前输出层还缺少明确的视觉反馈。用户侧期望是：
1. 若某 item 在当前 terminal mode / bandwidth 下不要求测试，则不计入未命中；
2. 输出 checklist 时，对应单元格应做灰色填充，和“真实漏测”区分开；
3. 该语义应同时体现在结果统计、详情弹窗和最终 checklist 中。

##### 17.1.3 不同模式下的适用性规则仍需继续收口
当前已确认的 mode / bandwidth 适用性规则如下：
1. Handsfree：仅 `NB` 网络不测 `Distortion - SND -4.7dBPa`。
2. Headset：`NB` 网络不测 `Distortion - SND -4.7dBPa`。
3. Headset：全部网络均不测 `STMR` 相关项。

后续不应再把这些差异散落在提取逻辑中，而应继续沉淀到 profile 规则配置中，包括：
1. `excludedTerminalModes`
2. `applicableBandwidths`
3. 必要时增加更高层的 `applicableNetworks`

##### 17.1.4 HE 当前最明显的问题不是单点漏提，而是“规则 + 位置 + 模板”未完全对齐
当前 Handsfree 至少存在以下已确认问题：
1. `Quality in the Presence of Ambient Noise` 相关 subitem 应填写具体数值，而不是 `PASS / FAIL` 之类的状态结论。这说明 HE 当前沿用了部分 HA 的 status 型规则，但 HE 对应位置实际需要 value 型提取。
2. `Echo control characteristics` 位于第 `28-32` 行，当前第 `28`、`29` 行未正确提取，需重点核查是：
	- 规则关键词未命中；
	- `outputCell` 与 HE 模板错位；
	- 或条目本身在 HE checklist 中与 HA 的 item 顺序不同。
3. HE 中“item 相同但 value 落点不同”的问题不能再按 Handset 规则直接平移，而应逐项对照 HE checklist，确认 item、subitem、outputCell 的真实映射关系。

因此，HE 当前问题不应理解成“再补几个关键词即可”，而应作为一次完整的 Handsfree profile 对表工作处理。

##### 17.1.5 HH 当前问题需要和“本来不测”显式分开
Headset 当前也不能简单沿用 HA 语义。除已确认的 `STMR` 全部不测外，后续还需继续区分两类情况：
1. 模板上根本没有对应测项，这类应标记为 `skip`，不能继续算失败；
2. 模板有该项，但报告源缺事实或规则未适配，这类才属于真实待修问题。

这一步的重点不是盲目追求 HH 命中数，而是先把“伪失败”和“真实失败”拆开，否则后续规则优化方向会持续失真。

##### 17.1.6 HH 当前已确认的专项问题清单
结合前面 `Utah_All` 的 Headset 样本核查，HH 当前应重点按“已确认不测 / 已确认修正 / 当前真实待处理”三层来管理，而不是继续把所有未命中项混成一类。

第一类，已确认不应继续按失败统计的项：
1. `STMR` 相关项在 Headset 全部网络下均不测，应直接纳入 skip，而不是继续作为提取失败项。
2. 若 Headset 模板上本身为空白、没有对应测项，则也应作为 skip 处理，不能继续沿用 Handset checklist 口径压到 HH 上。

第二类，已确认不应回退的修正项：
1. receiving sensitivity 三项已确认应匹配 `Sensitivity, frequency RCV Binaural MAX/NOM/MIN HHSB` 语义，后续不能再退回到 Handset 的 `char / 1/12oct / 1/3oct` 关键词假设。
2. item 35 `Delay - Round-trip - TS+TR` 已确认应锁定 `Calculation of Echo Delay + Calculated Value`，后续不能再让宽松 `Delay` 关键词把 sensitivity 的 `Min. dist. to tolerance scheme` 抢中。

第三类，当前仍需继续处理的真实问题：
1. item `10-15`：当前主要集中在 `Noise` 相关项。现阶段更需要先确认 HH 源报告里是否真的存在对应结构化事实，再决定是补规则还是维持未命中。
2. item `21-25`：这组里需要先拆出哪些已经属于“Headset 不测”语义，哪些才是真正需要提取的 sidetone / delay 项，避免把 skip 项继续混进规则修复列表。
3. item `34`：当前已能命中 `Distortion RCV -16 dBm0 activation NOM HHSB` 一类结果行，但源行提供的是“数值 + Done”，不是现有 `status_judge` 期望的 `OK / Not OK`。因此在业务口径明确前，不应把 `Done` 直接等价成 `Pass`。

当前对 HH 的处理原则应明确如下：
1. 先从模板与业务口径上剥离“本来不测”的项；
2. 再保护已经修正过的 Binaural sensitivity 与 Echo Delay 规则，不要被后续宽松兼容回退；
3. 最后只对真正仍存在事实且模板确实要求测试的项继续做规则优化。

#### 17.2 本轮建议的优化方案

##### 17.2.1 将 checklist 输出样式正式纳入 mode-specific template profile
建议将当前输出样式从“统一后处理”调整为“模板 profile 驱动”，至少包含以下配置：

1. 本来不测；
2. 应测但未提取到；
3. 已成功提取。

2. 再确认该 item 对应的是 value、status 还是 formula；
3. 再确认 subitem 顺序与 outputCell 落点；
4. 最后才补规则关键词、fallbackKeywords、exactRowKeywords。

也就是说，后续每修一项都至少要同时确认三件事：
1. 是否该测；
2. 填什么；
3. 填到哪里。

从当前问题集中度来看，建议优先级如下：
1. 先处理 HE 的 ambient noise value/status 错位、Echo control `28-32` 行映射以及模板样式问题；

### 19. 2026-03-19 checklist 写出链路改造记录

#### 19.1 改造背景
在继续核查 HE / HH 模板样式问题时，已确认当前输出文件样式失效的根因，不是单纯的 COM 样式脚本未执行，而是原有写出链路先用 `xlsx` 将整份 checklist 模板重写了一遍，导致模板内大量原始样式索引、边框和结构信息在写出阶段就已被打散。

直接对比 Headset 模板与输出文件内部结构后，可确认：
1. 原始模板在 `A3:K75` 区域内样式索引完整；
2. 原始输出文件中大量单元格缺少 `s` 样式属性；
3. 后续 COM 阶段只补数字格式和 skip 灰底，无法把已经丢失的模板样式再完整恢复回来。

因此，本轮正式将 checklist 写出主思路从“重写整本 workbook”调整为“先复制模板，再在复制件上做最小修改”。

#### 19.2 本轮代码改造

##### 19.2.1 checklist 主写出链路改为“复制模板优先”
修改文件：
1. `src/main/services/reportChecker/checklistWriter.js`
2. `src/main/services/reportChecker/excelComStyler.js`
3. `src/main/services/reportChecker/applyChecklistStyles.ps1`

本轮已将 checklist 写出逻辑拆成两层：
1. 先构建 `writePlan`，只计算需要更新的目标单元格、数字格式、skip 单元格和 Report 页控制位；
2. 主链路优先复制原始模板文件，再尝试在复制件上直接落值和落样式；
3. 仅在 COM 更新链不可用时，才退回旧的 `xlsx` 重写方案兜底。

这一步的目的不是简单“换个 API”，而是从根上避免整本模板被重写后导致原始样式、合并关系和动态表结构一起受损。

##### 19.2.2 Report 页动态切换入口已接入
本轮已确认当前 checklist 模板中，`Report` 页至少存在以下动态控制位：
1. `B15`：`Test Network`
2. `C15`：`Vocoder`

据此，当前写出层已接入基础的 Report 页更新能力：
1. 根据 `reportContext.network` 回填 `Report!B15`；
2. 根据 `reportContext.codec + bandwidth` 组合回填 `Report!C15`，例如 `EVS_SWB`、`AMR_NB`。

这意味着后续若模板继续沿用“同一份 checklist 内通过 Report 页切换 network / codec / bandwidth”的方案，当前主链已经具备基础承载能力，而不是只能静态复制后直接写 I 列。

##### 19.2.3 COM 直写链路已补充 payload 与分批更新机制
为避免一次性将大量更新项直接通过命令行参数塞给 PowerShell，本轮继续补充了以下工程化处理：
1. 更新项改为先写入临时 payload JSON 文件，再由 `applyChecklistStyles.ps1` 读取；
2. COM 更新被拆分为 `Report` 更新、数据单元格更新、样式更新三个阶段；
3. 数据单元格更新进一步按小批次分段执行，降低单次 COM 调用压力；
4. PowerShell 端对 `Range`、动态获取的 `Worksheet` 等 COM 对象增加了显式释放，减少对象泄漏导致的长时间挂起。

##### 19.2.4 旧链路暂时保留为 fallback
尽管当前主设计已切到“复制模板优先”，但本轮没有直接删除旧的 `xlsx` 重写链路，而是保留其作为 fallback，原因如下：
1. 当前环境下的 Excel / WPS COM 行为仍存在稳定性差异；
2. 若直接删除 fallback，一旦 COM 失败，整个 checklist 输出链会不可用；
3. 现阶段更重要的是先把新主思路稳定挂到主流程中，再逐步压缩 fallback 的触发面。

#### 19.3 本轮验证结果
本轮验证得到以下明确结论：
1. checklist 写出主链的架构方向已经切换为“复制模板后直写复制件”；
2. `Report` 页的动态切换入口已经接入当前主链，不再只是文档层方案；
3. 原始模板样式丢失的根因已确认来自旧的整本 `xlsx` 重写方案；
4. 最小化 COM 写入探针与 payload 机制已能单独执行，不再停留在纯设计阶段；
5. 但在完整 HE 样本上，第一段 COM 直写链路当前仍存在超时现象，导致流程在部分场景下仍会退回旧 fallback。

### 22. 2026-03-19 checklist 下拉参数解析修复与级联联动实现

#### 22.1 问题背景

用户反馈：上传 checklist 后任务参数面板（B13/B15/C15/D15）只显示单元格当前默认值，下拉候选项为空，无法手动切换。

根因：checklist 验证公式并非静态列表，而是命名区域 + INDIRECT 动态联动：
- B13 → `InterfaceTypes`（命名区域，指向 Report!$A$65:$C$65）
- B15 → `Networks`（命名区域，指向 Report!$A$45:$A$51）
- C15 → `INDIRECT($B$15&"_Vc_Bw")`（动态拼接命名区域名）
- D15 → `INDIRECT($C$15&"_Br")`（同上）

原解析器只支持静态逗号列表和显式 Sheet 区间，命名区域和 INDIRECT 直接返回空数组。

#### 22.2 本轮已完成内容

**`src/main/services/reportChecker/checklistReportPanel.js`**

1. **命名区域解析**：`resolveDefinedNameOptions` 通过 ExcelJS `workbook.definedNames.model` 按名称查找，将区域引用交给 `resolveSheetReferenceOptions` 取单元格值。
2. **INDIRECT 公式解析**：`evaluateIndirectExpression / resolveIndirectPart` 将表达式中的 `$B$15` 替换为单元格当前值或 cellOverrides 覆盖值，拼接后再走命名区域解析。
3. **cellOverrides 透传**：`extractOptionsFromValidation` → `resolveValidationFormulaOptions` → `evaluateIndirectExpression` → `resolveIndirectPart` 全链路透传，用于预计算时模拟任意上游值。
4. **cascadeMap 预计算**：`buildCascadeMaps` 在上传解析时一次性遍历所有 B15 候选值计算对应 C15 分支，再遍历所有 C15 候选值计算对应 D15 分支，以 `{ parentValue → [childOptions] }` 形式附在各字段上返回给前端。

**`src/renderer/pages/ReportChecker.jsx`**

1. 新增 `reportPanelDynamicOptions` state 存储当前动态候选项。
2. `updateReportPanelSelection` 重写为级联逻辑：改 B15 → 刷新 C15 候选并自动选第一项再级联刷新 D15；改 C15 → 刷新 D15 候选并自动选第一项。
3. Select 渲染优先消费 `reportPanelDynamicOptions[cell]`，不存在则回退静态 `options`。

### 25. 2026-03-19 18:29:14 输出文件无法打开修复（WPS）

#### 25.1 问题现象

- 用户反馈新生成 checklist 文件在 WPS 中提示“无法打开文件”。
- 受影响样例：`utah_HA_NB_checklist_20260319_1821.xlsx`。

#### 25.2 根因定位

- 根因是上一轮新增的“跨 sheet 公式缓存强制回写”逻辑对 worksheet XML 的 `<c>` 节点匹配过宽。
- 在包含自闭合单元格（`<c .../>`）的区域会写出非法片段（如 `/ t="str"`），导致 XML 结构损坏。
- 该问题会使 WPS/Excel 在加载工作簿时直接报错。

#### 25.3 修复策略

- 回滚高风险的“公式缓存强制改写”逻辑，仅保留安全链路：
	- Report 参数单元格值回写（B13/B15/C15/D15）；
	- 检测项输出单元格值回写。
- 保留并强化工作簿重算设置：在 `workbook.xml` 的 `calcPr` 写入
	`fullCalcOnLoad="1" calcOnSave="1" forceFullCalc="1"`，由表格软件在打开时按原公式联动刷新。

#### 25.4 验证结果

- 新生成样例：`utah_HA_NB_checklist_20260319_1825.xlsx`，可正常生成且可读取。
- 结构体检：未再发现此前损坏特征（例如 `/ t=`、异常嵌套 cell 片段）。
- 结论：已恢复输出文件可打开性，并保持模板原公式联动机制。
