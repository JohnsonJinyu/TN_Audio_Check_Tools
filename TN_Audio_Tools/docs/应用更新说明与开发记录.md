## 第一部分：功能说明

### 功能概述

应用更新功能用于保证 Windows 安装版桌面工具在国内外网络环境下都能完成以下动作：

- 应用内检查新版本
- 应用内下载更新安装包
- 下载完成后在应用内启动安装程序
- 当应用内下载失败或速度不理想时，切换到浏览器下载备用链路

当前版本的设计重点，不再是依赖单一 GitHub 自动更新元数据，而是将“版本检查”和“安装包下载”拆成两条可控链路，并分别做多源兜底。

### 当前能力边界

1. 当前仅支持 Windows 安装版在线更新。
2. 便携版仍不纳入在线更新链路。
3. 应用会优先尝试应用内下载，失败后可切换到浏览器下载。
4. 更新检查结果、下载状态、下载速度、当前下载源名称，都会显示在设置页的“版本更新”区域。

### 当前实现方案

#### 1. 版本检查

版本检查改为读取远程 `update-manifest.json`，并按多源顺序依次尝试：

1. Gitee raw
2. GitHub raw
3. jsDelivr

只要其中一个地址可用，应用就能拿到最新版本信息。

#### 2. 应用内下载

安装包下载不再只依赖单一地址，而是由 manifest 中提供多个下载源：

- 镜像加速源
- GitHub Release 直链

应用会先做快速探测，再优先使用更快的地址进行下载；如果当前源失败，则自动切换备用地址。

#### 3. 浏览器备用下载

若应用内下载失败，或者当前网络下速度仍不理想，用户可点击设置页中的“浏览器下载”按钮，直接打开备用下载地址。

#### 4. 安装动作

安装包下载完成后，应用会记录本地安装包路径；用户点击“重启安装”后，应用会直接启动安装程序并退出当前应用。

## 第二部分：本次专项中的坑与结论

这一部分只记录“应用更新”本身的坑，不与报告审查、测试数据收集等模块混写，便于后续独立追踪。

### 1. 旧方案的根本问题

旧方案基于 `electron-updater + GitHub publish metadata`。

表面上看已经加了镜像下载，但实际上镜像只覆盖了“下载安装包”，没有覆盖“检查更新”本身。

这会导致：

1. 国内网络下，只要 GitHub 元数据请求不稳定，就会在“检查更新”阶段直接失败。
2. 用户根本走不到下载阶段，镜像也就无法发挥作用。

结论：

- 单纯给下载地址套镜像，不能解决“检查更新失败”的问题。
- 必须把版本检查本身也迁移到国内外都能访问的多源方案上。

### 2. 中途方案里踩到的坑

#### 2.1 manifest 路径写错

中间版本曾出现“国内外都检查失败”的问题，根因不是网络，而是远程 manifest 地址写错。

实际可访问路径需要带上：

`TN_Audio_Tools/update-manifest.json`

如果路径少这一层目录，就会出现：

- Gitee 404
- GitHub raw 404
- jsDelivr 404

结论：

- raw 地址必须以远端真实仓库结构为准，不能只按本地工作区想象路径去拼。

#### 2.2 分支名判断错误

中间一度按 `main` 拼接远程清单地址，但实际远端生效分支是 `master`。

结论：

- 更新地址中的分支名不能想当然，必须与实际远端默认分支对齐。

#### 2.3 manifest 被拼接污染

发布 1.1.6 后，发现本地 `update-manifest.json` 被误拼成了两段 JSON 内容，导致文件虽然表面看“像有内容”，但实际上格式已经污染。

结论：

- update manifest 必须保持单一、完整、可直接反序列化的 JSON。
- 发布前应单独校验该文件，而不能只看构建是否成功。

#### 2.4 网页抓取结果可能有缓存

在验证远程 manifest 时，直接抓 raw 页面返回的内容一度仍显示旧版本，和实际 GitHub API 返回结果不一致。

结论：

- 验证远程 manifest 时，优先用 GitHub API 或分支 SHA 校验，不要只依赖网页抓取快照。

#### 2.5 临时构建目录容易被运行中的应用占用

临时输出目录 `dist-release-1.1.5` 中的 `app.asar` 会被正在运行的应用实例占用，导致目录无法删除。

结论：

- 临时构建目录清理不应与版本发布强绑定。
- 如果要删临时目录，先关闭正在运行的桌面应用。

### 3. 本次最终解决方案

当前最终方案为：

1. 用远程 `update-manifest.json` 做版本检查。
2. manifest 提供多个下载源。
3. 应用内下载前对多个下载源做快速探测与排序。
4. 应用内下载失败时，可自动或手动切换到浏览器下载。
5. GitHub Release 继续作为正式安装包发布面。
6. Gitee / GitHub raw / jsDelivr 共同承担清单分发。

这个方案的优点是：

1. 检查更新不再依赖单一 GitHub 元数据。
2. 下载更新不再依赖单一下载源。
3. 国内外网络都具备可用路径。
4. 浏览器下载可以作为最后兜底，不会把更新能力卡死在应用内。

## 第四部分：后续注意事项

1. 每次发版时，必须同步更新：
   - `package.json`
   - `package-lock.json`
   - `update-manifest.json`
   - GitHub Release 资产
2. 每次发版后，至少做两类验证：
   - 应用内“检查更新”是否能正常拿到 manifest
   - GitHub Release 附件是否与 manifest 中的版本和文件名一致
3. 如果后续更换镜像下载源，优先改 `update-manifest.json`，尽量不要再改客户端代码。
4. 若后续要进一步提升体验，优先考虑：
   - 下载失败原因展示更明确
   - 下载速度展示更友好
   - 发布清单生成与发布动作进一步自动化
